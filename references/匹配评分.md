# 匹配评分

用于按 JD 需求画像给 GitHub 候选项目打匹配分、算 JD 覆盖矩阵、选短名单。

## 评分模型

`search_jd_candidates.py` 已内置基础打分，最终取舍仍要人工按下面规则复核。

```text
匹配分 = 技术栈重合度（每命中一个 JD 硬技术栈 +3）
       + 业务域吻合度（每命中一个 JD 业务域 +2）
       + 主语言吻合（+3）
       + 社区验证弱加分（1k+ star +2，封顶）
       + 描述完整（+1）
       - 框架/工具/脚手架降权（命中 framework/sdk/library/template/demo 等）
       - 过热项目轻微降权（20k+ star -1）
       - IoT/硬件降权（JD 未要求时 -5）
```

## JD 覆盖矩阵

每个候选都要算覆盖矩阵，用于短名单展示和最终匹配度报告：

- `技术覆盖`：命中的 JD 硬技术栈数 / JD 硬技术栈总数。
- `业务覆盖`：命中的 JD 业务域数 / JD 业务域总数。
- `语言吻合`：项目主语言是否等于 JD 主语言。

覆盖只来自描述 / topics / README probe，属于初筛预估，不能当成最终「负责功能」的证据。

## 排序与取舍

1. 主排序按匹配分（技术栈重合 + 业务域吻合），不是按 star。
2. 技术覆盖和业务覆盖都高的优先；只命中业务域、技术栈完全不沾的项目降权。
3. 分接近时，优先业务闭环清晰、源码可验证、可二次改造的项目。
4. 短名单尽量跨语言 / 跨细分场景，避免 3 个项目都是同一类商城复刻。
5. 对反复出现的学生简历热门项目（普通商城、外卖、网盘克隆）做降权，除非它具备订单状态机、支付回调幂等、库存预占、对账补偿、搜索索引等业务深度。

## 硬过滤

命中即淘汰：

- awesome / tutorial / boilerplate / storefront / ui kit / 浏览器插件 / 桌面壳 / 脚手架模板。
- 只调用一次大模型并展示结果的薄封装。
- 纯框架 / SDK / library / 工具集 / prompt 集合 / 论文复现脚手架。
- 无业务数据、状态流转、持久化、失败恢复的项目。
- IoT / 嵌入式 / 硬件接入 / 设备管理 / 工业控制（JD 未要求时）。

不要因为是电商 / 网盘 / 社区 / 工单 / CRM 题材就淘汰；只淘汰浅层 CRUD 和讲不清业务链路的复刻。

## 与 JD 对齐的深挖方向

为了让简历条目对得上 JD，从源码里优先找与 JD 职责动词相关的机制：

- JD 写「高并发 / 高性能」→ 找缓存多级、热点 key、限流、线程池、异步编排。
- JD 写「一致性 / 数据准确」→ 找分布式事务、本地消息表 / Outbox、对账补偿、幂等。
- JD 写「交易 / 订单 / 支付」→ 找订单状态机、支付回调幂等、库存预占与回滚、超时取消。
- JD 写「检索 / 搜索」→ 找索引设计、增量同步、搜索一致性。
- JD 写「权限 / 安全」→ 找 RBAC、OAuth2 / 网关鉴权、接口签名、审计日志。
- JD 写「Agent / RAG / 大模型」→ 找工具调用、检索链路、记忆、评测、人类审批、任务编排。

每个推荐项目至少挖 5-6 个能体现工程难度、且与 JD 沾边的负责功能，且都要有本地源码证据。
