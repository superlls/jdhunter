# 简历写法

## 核心原则

简历写法必须分成三部分：

1. `项目简介`：一段 80-120 字左右的自然语言介绍。
2. `负责功能 / 技术难点`：5-6 条有技术难度的负责功能，必须像真实简历一样写清业务问题、技术方案和异常处理。
3. `建议简历功能点（完成对应改造后可写）`：如果用户尚未实现，只能作为建议写法输出，必须说明完成对应改造后才能写。

不要把简历写法开头写成 `项目名｜项目类型｜技术栈` 这种关键词列表。用户需要的是能直接放进简历项目介绍里的自然语言。

项目简介要回答：

- 这个项目基于什么背景或开源项目。
- 搭建了什么业务系统。
- 模拟或解决了什么真实业务流程。
- 最终产出了什么业务价值。

示例：

```text
本项目基于 TradingAgents 搭建多智能体 AI 投研与模拟交易决策平台，模拟真实投研团队的协作流程，让不同 Agent 分别完成市场分析、观点辩论、交易建议、风险评估和组合审批，最终生成可追溯的结构化投研报告，用于辅助投资研究和策略复盘。
```

简历功能点要同时体现业务思考和技术落地。功能点默认是“建议写法”，不是仓库已有成果；除非用户明确说已经实现，或本轮确实在本地完成了代码改造，否则不能用“已完成 / 已实现 / 负责上线”口吻包装。推荐写成：

```text
基于 X 实现 Y 功能，解决 Z 边缘问题 / 性能问题 / 一致性问题。
```

## 技术难度表达标准

功能点不能停留在“实现某功能、接入某组件、提供某接口”的层面，必须挖出能体现工程难度的机制。每条负责功能必须先经过本地源码、配置、测试、迁移脚本或运行入口证据核验，再组织成自然的简历表达；下面是必须覆盖的表达要素，不是固定句式：

```text
【具体业务动作 / 模块】 + 【核心技术机制】 + 【关键实现细节】 + 【异常或边界处理】 + 【可追问结果】。
```

禁止把所有条目写成同一种开头或同一种语序，尤其不要连续使用“围绕……”“基于……”“通过……”这类模板化开头。5-6 条负责功能中，相同开头动词或相同句式最多出现 1 次；每条都要根据源码证据选择更贴近业务动作的表达。

强表达必须满足至少 3 个条件：

- 有明确业务目标：例如接口使用率统计、调用链路追踪、工单流转、告警收敛、权限同步、知识库更新、策略回测。
- 有关键技术机制：例如 DTS / binlog 增量同步、MQ 重试补偿、延迟消息、双缓存、本地缓存上报缓存中心、CompletableFuture 异步编排、线程池隔离、Intern / 字典池、状态机、规则引擎、索引增量构建、分布式锁、幂等表。
- 有工程问题：例如数据一致性、消费堆积、缓存脏读、重复创建对象、父子线程池死锁、重复通知、越权访问、任务失败恢复、接口幂等。
- 有边界处理：失败重试、补偿任务、幂等消费、去重、限流、降级、死信队列、对账、超时控制、审计日志。
- 有可追问对象：能讲清数据表、消息 Topic、任务状态、缓存 Key、线程池策略、索引字段、权限模型、状态流转或评测指标。

## 负责功能写法模板

优先输出接近下面这种力度的句子，而不是泛泛的功能清单。示例展示表达强度和句式多样性，不要求照抄技术名词：

```text
1. 统计新增接口使用率：定时聚合埋点调用数据，并订阅 binlog 将业务表增量同步到 ES；同步失败时引入 MQ 重试和补偿任务，保证统计口径最终一致。
2. 搭建业务线级调用计数器：用本地缓存承接高频写入、分布式缓存保存全局计数，再由定时任务批量上报缓存中心；峰值流量下用延迟消息削峰，降低消费堆积风险。
3. 优化接口缓存对象创建链路：将同类业务接口抽象为工厂管理的单例缓存器，并对高频字符串入参做 Intern / 字典池处理，减少重复对象创建和 GC 压力。
4. 拆分下游接口溯源查询：按调用日志、接口版本和业务方标识生成追踪链路，使用 CompletableFuture 并行拉取子任务并隔离线程池，规避父子任务共用线程池导致的死锁。
5. 收敛工单状态流转入口：把提交、审批、驳回、重开等动作统一落到状态机校验，结合版本号和审计日志处理重复提交、越权审批和异常回滚。
```

以上是表达力度示例；必须结合项目真实场景选择合适机制，并主动改变句子开头、主语和语序，避免输出一组看起来由同一模板批量生成的条目。

## 技术深度挖掘方向

为每个项目找功能点时，优先从下面方向挖掘，而不是只写产品功能。不要机械堆词，必须结合项目业务场景选择能讲清实现细节的技术 key。

- 数据同步与一致性：binlog / CDC / DTS、Outbox、本地消息表、事务消息、TCC / Saga、补偿事务、对账任务、最终一致性、读写一致性、索引一致性、缓存一致性。
- 消息队列与异步链路：MQ 重试补偿、延迟消息、死信队列、幂等消费、顺序消息、事务消息、消息去重、消费位点、消费堆积治理、削峰填谷、异步回调、事件驱动架构。
- 缓存与高并发读写：本地缓存 + 分布式缓存双缓存、多级缓存、热点 Key、缓存预热、缓存穿透 / 击穿 / 雪崩、逻辑过期、互斥重建、BloomFilter、批量合并、缓存旁路、写穿 / 写回策略。
- 并发控制与幂等：分布式锁、乐观锁、悲观锁、版本号、CAS、自旋重试、防重复提交、幂等 Token、幂等表、唯一索引防重、接口幂等、库存 / 配额扣减、防重放。
- 任务调度与后台执行：定时任务、分片任务、分布式调度、任务状态机、任务抢占、任务重入、失败恢复、超时控制、重跑机制、任务租约、心跳续约、调度漂移治理。
- 线程池与异步编排：CompletableFuture、线程池隔离、异步聚合、父子任务死锁规避、队列容量治理、拒绝策略、超时熔断、背压、上下文透传、异步异常收敛。
- 流量治理与稳定性：限流、熔断、降级、隔离舱、灰度发布、开关配置、预案降级、重试退避、超时控制、舱壁模式、热点隔离、接口保护、容量评估。
- 领域建模与事件驱动：DDD、聚合根、领域事件、事件总线、Outbox、CQRS、事件溯源、Saga、补偿流程、读模型 / 写模型拆分。
- 数据库与存储优化：索引设计、联合索引、覆盖索引、分页优化、慢 SQL 治理、读写分离、分库分表、分片键设计、冷热数据分层、归档任务、批量写入、连接池调优。
- 检索与索引系统：ES 索引设计、增量更新、别名切换、零停机重建索引、召回字段设计、权重排序、分词策略、查询 DSL 优化、深分页治理、索引一致性、搜索降级。
- 权限与安全：RBAC / ABAC、租户隔离、数据权限、Token 刷新与失效、JWT 黑名单、OAuth2 / OIDC、接口签名、请求防重放、审计日志、越权检测、敏感字段脱敏。
- 可观测性与排障：TraceId、链路追踪、指标埋点、结构化日志、日志检索、告警聚合、告警去重、告警收敛、SLA / SLO、错误码治理、慢调用分析、业务看板。
- 接口治理与网关：接口版本管理、接口编排、聚合接口、网关鉴权、动态路由、限流插件、灰度路由、接口溯源、调用方识别、API 兼容性、下游依赖降级。
- 规则引擎与状态流转：状态机、规则引擎、策略模式、责任链、审批流、条件路由、规则热更新、规则命中日志、状态补偿、状态幂等、状态回滚。
- 接口接入与协议：Webhook、SSE、gRPC、WebSocket、长轮询、回调签名、幂等回调、接口重放保护、第三方鉴权。
- 配置与发布治理：Feature Flag、配置热更新、蓝绿发布、金丝雀发布、预案切换、版本回滚、渐进式开关、服务发现、优雅停机。
- 安全与合规控制：KMS、密钥轮换、字段加密、传输加密、脱敏策略、访问审计、操作留痕、风控规则、黑白名单、敏感动作二次确认。
- 验证与稳定性演练：压测、基准测试、故障演练、混沌工程、回放测试、回归评测、容量评估、降级预案验证、链路压测。
- 数据处理与指标体系：ETL、实时 / 离线同步、窗口聚合、去重聚合、指标口径治理、维度建模、宽表、埋点采集、数据补齐、离线重算。
- 对象与内存优化：对象复用、工厂 / 单例、字符串常量池、Intern / 字典池、批处理、连接池、对象池、大对象拆分、内存泄漏排查、资源释放、序列化优化。
- 文件与大数据处理：分片上传、断点续传、秒传、预签名 URL、文件去重、异步转码、批量导入、导入校验、失败行回写、流式处理、临时文件清理。
- 实时通信与协作：WebSocket、长连接心跳、在线状态、消息 ACK、离线消息、未读计数、消息漫游、会话同步、连接重连、广播与单播、顺序性保证。
- 支付 / 订单 / 交易：订单状态机、支付回调幂等、超时关单、库存预占、优惠券锁定、退款补偿、对账、分布式事务、金额精度、重复支付防护。
- 推荐 / 内容 / 社区：Feed 流、推拉结合、时间线合并、点赞计数一致性、评论楼中楼、内容审核、热榜衰减、曝光去重、召回与排序、反作弊。
- Agent 工程：工具调用状态机、任务规划 DAG、长期记忆与短期记忆、上下文压缩、RAG 召回评测、向量索引、引用溯源、人工审批、执行沙箱、工具超时、失败回滚、多 Agent 协作、工具路由、guardrails、trace replay、evaluation harness。

## 简历功能点输出约束

- 标题必须使用 `建议简历功能点（完成对应改造后可写）`，不要直接写成 `简历功能点`，避免误导用户把未实现内容当成已有经历。
- `负责功能 / 技术难点` 至少输出 5 条，优先输出 6 条；少于 5 条说明挖掘不足，必须继续阅读代码或换项目。
- 每条负责功能必须能在“代码验证摘要”中找到对应本地源码依据；本地源码必须来自 `pull_github_repos.py` 成功拉取的仓库。简历条目本身不要写“证据：代码已验证 / 来源：文件”这类前缀；要按真实简历口吻直接写业务问题、技术方案、异常处理和工程收益。
- 每条功能点只写 1 个明确业务场景 + 1 个核心技术动作 + 1 个可追问结果，不要堆 4 个以上技术名词。
- 每条功能点必须体现技术难点；如果只能写“实现了页面 / 接入了模型 / 提供了接口”，说明挖掘不够，必须继续追问数据流、并发、缓存、一致性、异步、权限、可观测性或失败恢复。
- 每个项目的 5-6 条负责功能必须做句式去重：不能连续使用同一个开头词、同一个“业务目标 + 设计机制 + 解决问题”模板，尤其不能批量写成“围绕 XXX ...”。
- 功能点必须落在当前项目能自然承载的模块上，不能凭空添加仓库没有基础、用户也没有实现的复杂能力。
- 如果功能点来自 `建议二次改造`，用“设计 / 增加 / 改造 / 接入 / 构建”描述可做事项，不要写“已上线 / 已落地 / 已保障”。
- 如果功能点来自 `已有能力`，可以写“基于仓库已有 X 能力，二次改造 Y”，但仍要说明用户需要完成的改造动作。
- 每个功能点要能被追问到实现细节，例如数据表、接口、任务状态、失败重试、幂等、权限、缓存、索引、告警、评测指标。
- 不要写空泛收益，例如“提升系统性能”“增强用户体验”；必须说明通过什么机制解决了什么具体问题。

## 功能点自检清单

生成每条功能点前，必须逐条自检：

1. 这是仓库已有能力，还是建议改造？
2. 这条功能点的证据来自源码、配置、测试、迁移脚本、运行入口，还是建议改造？
3. “代码验证摘要”是否能指出脚本 manifest、本地仓库目录、commit、模块、目录、类名、配置或测试用例？如果不能，不能写成负责功能；但正式简历条目不要内嵌源码路径或证据标签。
4. 用户如果没实现，是否会造成简历造假？
5. 这条能否说清业务场景、技术方案和异常处理？
6. 有没有把配置、部署、接 API 包装成核心亮点？
7. 有没有把多个无关技术点硬塞进一条？
8. 这条有没有体现至少一个工程难题，如一致性、并发、性能、异步、幂等、权限、可追踪、失败恢复？

## 弱表达与强表达

弱表达：

- 使用通用后端技术完成系统开发。
- 将数据同步到检索引擎，加快查询速度。
- 将数据同步到缓存，加快查询速度。

强表达：

- 基于二进制日志监听实现关系型数据库到全文检索引擎的增量同步，在同步失败时通过消息队列重试和补偿任务保障最终一致性。
- 采用本地缓存 + 分布式缓存 + 数据库的多级缓存链路承接高频查询，并通过定时对账降低缓存脏读风险。

## 避免配置类亮点

不要把下面这些当成核心项目亮点：

- 使用配置中心。
- 配置反向代理。
- 使用容器部署。

只有当这些动作解决了明确工程问题时，才值得写进简历。

## 新奇项目写法

新奇项目重点突出：

- 核心场景：谁在什么场景下使用它。
- 产品巧思：它为什么比普通项目更亮眼。
- 功能落地：做了什么具体能力。
- 使用效果：提升效率、降低操作成本、覆盖某类工作流。

智能体项目可重点写：

- 工具调用
- 长期记忆与上下文压缩
- 任务规划与人工确认
- 检索增强生成
- 沙箱执行
- 评测集
- 多智能体协作
- 工作流编排

## 扎实项目写法

扎实项目要主动暴露可追问点：

- 缓存：多级缓存、缓存穿透 / 击穿 / 雪崩、热键、本地缓存一致性。
- 消息队列：异步削峰、延迟队列、失败重试、死信队列、幂等消费。
- 数据一致性：二进制日志、对账、补偿任务、最终一致性。
- 并发：分布式锁、乐观锁、库存扣减、防重复提交。
- 搜索：索引设计、增量同步、搜索性能、索引一致性。
- 可观测性：链路追踪、指标、日志检索、告警。
- 权限：角色权限、单点登录、接口签名、网关鉴权。

每个技术点都要回答：

1. 业务场景是什么？
2. 为什么需要这个技术？
3. 解决了什么边缘问题？
4. 如果不用它会有什么问题？
5. 如果被问到实现细节，能不能讲清原理和取舍？

## 可信度边界

必须区分：

- `已有能力`：只能指经过本地源码、配置、测试、迁移脚本或运行入口验证的能力；未完成本地证据核验的 README / 文档描述不得写入“负责功能 / 技术难点”。
- `建议改造`：为了让项目更适合简历而建议补充的能力。
- `可写入简历`：用户真正实现或准备实现后才能写的内容。

输出时必须额外区分：

- `代码已验证`：已经通过 `pull_github_repos.py` 成功拉取本地仓库，并阅读本地仓库中的关键代码、目录、类、配置、迁移脚本、运行入口或测试，确认功能确实存在；证据统一写在“代码验证摘要”，不要逐条塞进简历功能点。
- `README / 文档已验证`：只能用于项目定位和候选池说明，不能支撑“负责功能 / 技术难点”，也不能替代本地代码验证。
- `建议改造`：当前仓库源码未确认存在，是为了增强简历价值建议用户实现。

不要把建议改造包装成已经完成的成果。

## 输出边界

默认只输出项目简介、负责功能 / 技术难点、建议简历功能点、可改造方向和落地计划。除非用户明确要求，否则不要额外生成问答模块。
