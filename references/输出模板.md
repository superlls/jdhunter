# 输出模板

除非用户指定其他名字，在当前工作区生成 `jd-match-resume-pack.md`，结构如下：

```markdown
# JD 匹配项目推荐与简历写法

> 生成日期：YYYY-MM-DD
> 目标岗位：从 JD 解析出的岗位名 + 经验年限
> JD 来源：用户粘贴的 JD 文本（不联网抓取）
> 可信度边界：负责功能必须来自 `pull_github_repos.py` 拉取到本地的源码证据，证据统一放在“代码验证摘要”。JD 覆盖矩阵只来自描述 / README probe，属于初筛预估，不能支撑负责功能。代码已验证和建议改造要分开写。

## 结论先行

- 推荐组合：每个项目一句话说明它命中了这份 JD 的哪些硬要求。
- 匹配逻辑：为什么这些项目对该 JD 匹配度最高。
- 定位分工：...

## JD 需求画像（解析回显）

- 岗位 / 经验：...
- 硬技术栈：...
- 加分技术：...
- 业务域：...
- 职责重点：...

## JD 匹配度报告

逐条列出 JD 的关键要求，标注由哪个项目、哪段功能命中，哪些需改造后才能覆盖。

| JD 要求 | 类型(硬技术栈/业务域/职责) | 命中项目 | 命中点 | 覆盖状态(已覆盖/需改造/未覆盖) |

## 多样性说明

- 搜索覆盖的技术栈 / 业务域查询：...

- 如果不想做方向 B，可以换成：...

- 对该 JD 的匹配点：命中了 JD 的哪些硬技术栈 / 业务域 / 职责。
- 链接：...（必须输出真实来源地址，不允许胡编乱造）
- 适合人群：...
- 已有能力：...
- 代码验证摘要：必须列出 `pull_github_repos.py` manifest 路径、本地仓库目录、commit，以及已阅读的关键目录 / 文件 / 类 / 配置 / 测试；未完成脚本拉取和本地代码验证的项目不得进入最终推荐。
- 建议二次改造：优先补 JD 要求但项目尚缺的能力。

### 简历写法

#### 项目简介

用一段 80-120 字的自然语言介绍项目，不要写成 `项目名｜类型｜技术栈` 的关键词堆叠。要说明：基于什么项目 / 业务背景、搭建了什么系统、解决了什么真实流程、产出什么业务价值。

#### 负责功能 / 技术难点
(5-6 条，必须来自源码验证；尽量复用 JD 的关键词与职责动词，但不得为贴合 JD 而编造；句式去重，禁止批量写成“围绕 XXX ...”)
- 1. 【业务动作 / 模块】：【核心技术机制】+【关键实现细节】+【异常或边界处理】+【可追问结果】。
- 2. 【换主语和语序】：讲数据流 / 调用链路 / 状态流转中的工程难点，写清失败重试、补偿、幂等、降级、权限或可观测性。
- 3. ...
- 4. ...
- 5. ...
- 6. ...

#### 建议简历功能点（完成对应改造后可写）

- 优先写能补齐 JD 要求、但项目当前没有的能力，并标明是改造后才可写。
- ...

### 可改造方向（对齐 JD）

- 列出为了更贴合这份 JD，建议补的改造点。

## 推荐项目 2：项目名

...（同上结构）

## 最终建议

- 哪个项目优先做、为什么最贴合该 JD。
- 投递这份 JD 时简历该如何排列这些项目。
```

## 通用输出要求

- 尽量只使用中文。
- 每个推荐仓库都要给出真实链接。
- 清楚区分已有能力、建议改造、可写入简历。
- 必须输出代码验证摘要；摘要必须包含 `pull_github_repos.py` manifest 路径、本地仓库目录和 commit。负责功能必须来自本地源码证据；没有脚本拉取记录或本地源码证据的项目不得进入最终推荐；正式简历条目不要逐条写源码路径或证据标签。
- 简历功能点默认标注为“建议简历功能点（完成对应改造后可写）”，不能把建议改造写成已完成成果。
- 负责功能要写出工程难点、并尽量对上 JD 职责，不能只写产品动作；每个项目至少 5 条、优先 6 条。
- 同一项目内不得反复使用“围绕 / 基于 / 通过”等相同开头或语序。
- JD 匹配度报告里凡是「需改造 / 未覆盖」的要求，不能在简历正式负责功能里写成已实现。
- 默认排除 IoT / 硬件方向，除非 JD 明确要求。
- 不要大段复制说明文档原文。
- 保持推荐适合在校生或早期候选人。
